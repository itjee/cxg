**문제**: 일부 boolean이 `is_/has_/should_/can_` 접두사 없음

```sql
❌ carryover_allowed    → ✅ is_carryover_allowed
❌ auto_renew          → ✅ is_auto_renew
❌ manual_entry        → ✅ is_manual_entry
❌ require_approval    → ✅ is_require_approval
```

**노력**: 2-4시간
**위험**: 낮음
**우선순위**: Medium (읽기 명확성)

### Issue 2: Temporal 접미사 (2개 컬럼, 0.07%)

**문제**: TIMESTAMP인데 `_date` 접미사 사용

```sql
❌ response_date (TIMESTAMP)  → ✅ response_at
❌ approval_date (TIMESTAMP)  → ✅ approval_at
```

**규칙**:
- DATE 타입 → `*_date`
- TIMESTAMP 타입 → `*_at`

**노력**: 30분
**위험**: 매우 낮음
**우선순위**: Low (기술적 정확성)

## 💡 왜 ConexGrow가 옳은가?

### vs Stripe/Shopify 스타일

```
Stripe (API 최적화):
❌ "customer_id" → "customerId" (camelCase)
→ JSON 응답 크기 최소화
→ API 중심 설계

ConexGrow (데이터베이스 중심):
✅ "customer_id" (snake_case)
→ SQL 쿼리 가독성 최대화
→ 데이터 무결성 우선

장점:
- 내부: snake_case (SQL 최적)
- 외부: camelCase로 변환 (JSON/API 최적)
- 양쪽 이점 취할 수 있음
```

### ERP 시스템의 정석

```
SAP, Oracle, NetSuite도 동일:
✅ snake_case 데이터베이스
✅ UUID 또는 surrogate key
✅ NUMERIC 금액
✅ 100% 감사 추적
```
